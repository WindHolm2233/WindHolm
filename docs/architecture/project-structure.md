# 项目结构说明

## 目标

本次目录整理遵循软件工程里的几个基本原则：

- 按职责拆分目录，减少“所有文件都堆在一起”的情况
- 让配置、工具、内容、组件的边界更清晰
- 在不破坏现有页面路由的前提下，提高后续扩展和维护效率
- 让项目更适合作为课程设计、竞赛作品或毕业设计展示

## 当前推荐结构

```text
src/
├── components/         # 视图组件，按职责拆分
├── config/             # 业务与界面配置
│   ├── core/           # 站点核心配置
│   ├── layout/         # 布局类配置
│   ├── theme/          # 主题与样式配置
│   ├── features/       # 业务功能配置
│   ├── integrations/   # 第三方集成配置
│   └── index.ts        # 统一导出入口
├── constants/          # 常量定义
├── content/            # Astro 内容集合与文章
├── i18n/               # 国际化资源
├── layouts/            # 页面布局
├── pages/              # 路由页面
├── plugins/            # Markdown / Astro 插件
├── scripts/            # 前端运行期脚本
├── styles/             # 全局样式与主题样式
├── types/              # 类型定义
└── utils/              # 工具函数
    ├── content/        # 内容处理相关
    ├── layout/         # 布局与侧边栏相关
    ├── navigation/     # 路由与链接相关
    ├── system/         # 通用系统能力
    ├── theme/          # 主题与外观切换
    └── dev/            # 调试辅助文件
```

## 配置分层建议

- `core/`：站点基础信息、个人资料等全局核心配置
- `layout/`：导航栏、侧边栏、页脚等布局相关配置
- `theme/`：字体、代码高亮、樱花特效等视觉层配置
- `features/`：公告、许可证、广告位等站点功能配置
- `integrations/`：评论系统、音乐播放器、Pio、友链等集成配置

## 工具层分组建议

- `content/`：文章、日期、Markdown 辅助处理
- `layout/`：布局计算、侧边栏控制、组件映射
- `navigation/`：URL 拼接、页面跳转、路径处理
- `system/`：图标、语言等底层通用能力
- `theme/`：主题切换、壁纸模式、动效管理

## 为什么这样更适合毕设

- 目录语义更清楚，老师或答辩成员能快速看懂项目组织方式
- 后续补写设计文档时，容易从“系统配置层、工具层、表现层”角度展开说明
- 新增功能时更容易判断代码应该放在哪，减少结构继续变乱
- 更接近真实项目中的模块边界，而不是单纯“按文件类型堆放”

## 后续可以继续优化的方向

- 为 `components/` 继续补一层按业务域拆分的结构
- 为 `content/posts/` 增加更明确的分类策略与命名规范
- 为 `docs/` 补充系统架构图、数据流图、部署说明和答辩用材料
- 为 `src/types/` 进一步区分配置类型、内容类型、组件类型

## 工程化建议

- 日常检查可使用 `npm run type-check`
- Astro 组件与内容校验可使用 `npm run check`
- 完整交付前可使用 `npm run verify`

这套流程更适合作为毕设阶段的“可验证开发过程”，方便在答辩时展示项目不仅能运行，也有基本的工程质量控制。
